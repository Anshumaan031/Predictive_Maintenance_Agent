# Changes — 2026-07-14

## 1. `/user <id>` switching

**Files:** `src/agent/cli.py`, `src/api/app.py`, `src/api/models.py`, `src/agent/prompts.py`

Added the ability to switch the active user identity at runtime without restarting the agent.

- **CLI** — new `/user <id>` slash command updates `identity.owner_id`, clears conversation history and active machine context. Because `MemoryService` holds a reference to the `Identity` dataclass, memory search and store immediately scope to the new owner with no restart.
- **API** — new `POST /session/user` endpoint (`{"owner_id": "..."}`) does the same and returns an updated `SessionInfo`.
- **Help text** updated to document the new command.

Demo use: start as `machine-floor`, ask about M104, then `/user operator-2` — the pre-seeded M104 memories won't appear because they are scoped to `machine-floor`.

## 2. `OWNER_ID` / `SESSION_ID` env vars

**Files:** `src/agent/config.py`, `src/agent/cli.py`, `src/api/state.py`, `.env`

Moved the hardcoded demo identity out of source code into optional environment variables.

- `DEFAULT_OWNER_ID` and `DEFAULT_SESSION_ID` constants are now defined once in `src/agent/config.py` (was duplicated in `cli.py` and `api/state.py`).
- `Settings` dataclass gained `owner_id` and `session_id` fields, read from `OWNER_ID` / `SESSION_ID` env vars with the original hardcoded values as fallbacks.
- Both `cli.py` and `api/state.py` now read identity from `settings` instead of local constants.
- `.env` updated with commented-out `OWNER_ID=machine-floor` / `SESSION_ID=session-1` so the knobs are discoverable.

To run against a different dataset, set `OWNER_ID=my-dataset` in `.env` — no source edits needed.

## 3. Streaming responses (CLI + API)

**Files:** `src/agent/cli.py`, `src/api/app.py`

Replaced the blocking `agent.run` call with `agent.run_stream_events` in both the CLI chat loop and the API SSE handler.

### Event handling

`run_stream_events` yields a sequential stream of typed events. Both implementations handle the same four:

| Event | Action |
|---|---|
| `FunctionToolCallEvent` | Print / emit tool call with name + args as it fires |
| `PartStartEvent(TextPart)` | First text chunk — print header (`iris ›` in CLI, `token` SSE event in API) |
| `PartDeltaEvent(TextPartDelta)` | Subsequent text chunks — stream immediately |
| `AgentRunResultEvent` | Capture full output, update history, log to memory |

Tool calls fire before text starts (that is the natural ordering of the event stream), so the `↳ tool` lines still appear above the response.

### CLI

- Removed the `console.status` spinner.
- Tool call lines print as each tool fires.
- `iris ›` header prints on the first text chunk (`PartStartEvent` with `TextPart`).
- Text chunks written with `print(..., end="", flush=True)` (bypasses Rich's per-call processing overhead for fine-grained chunks).
- Removed `_render_tool_calls` helper (logic now inline in event loop).
- Removed unused `ToolCallPart` and `Markdown` imports.

### API

- `_chat_sse` now emits `{"type": "token", "content": "..."}` events for each text delta — clients can render tokens as they arrive.
- Tool call SSE events now fire as tools complete rather than being batched after the full response.
- Final `{"type": "text", ...}` and `{"type": "done", ...}` events preserved — existing clients that only listen for these continue to work unchanged.
- `chat_sync` (`POST /chat/sync`) intentionally left on `agent.run` — it is a Swagger/Postman testing endpoint and blocking behaviour is correct there.
